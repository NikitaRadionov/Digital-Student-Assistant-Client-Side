## ЧАСТЬ 1 — ФУНДАМЕНТАЛЬНЫЕ КОНЦЕПЦИИ

### Что такое Use Case

Use Case — **контракт о поведении системы** между всеми stakeholders.  
Каждый шаг обязан быть одним из трёх:
- **Взаимодействие** двух акторов (передача данных или запроса)
- **Валидация** (система защищает интерес stakeholder)
- **Внутреннее изменение состояния** (система обновляет от имени stakeholder)

Use Case — это глава 3 требований, не всё. Вне него остаются: бизнес-правила, форматы данных, NFR, внешние интерфейсы.

---

### Уровни целей (Goal Levels)

| Символ | Цвет | Уровень | Критерий |
|--------|------|---------|----------|
| ☁️ cloud / 🪁 kite | Белый | Summary | Объединяет несколько user goals; несколько сессий/дней |
| 🌊 waves | **Синий** | **User Goal (sea-level)** | Одна сессия (2–20 мин); тест кофе-паузы ✓ |
| 🐟 fish | Индиго | Subfunction | Поддерживает user goal; нужна только если сложная или переиспользуется |
| 🐚 clam | Чёрный | Too Low | Не оформлять в отдельный UC — влить в родительский |

**Тест кофе-паузы:** «Можно ли уйти на кофе ПОСЛЕ этой цели?» → Да = sea-level.  
**Тест сотрудника:** «Засчитывается ли как выполненная работа?» → Да = sea-level.  
**Техника "Почему?":** Если UC слишком низко — спроси «Зачем актор это делает?» — ответ и есть уровень выше.  
**Правило 3–9:** Хорошо написанный user goal UC имеет 3–9 шагов в main success scenario.

---

### Акторы и Stakeholders

- **Primary Actor** — тот, у кого есть цель; инициирует UC.
- **Supporting Actor** — внешняя система или человек, которые предоставляют сервис.
- **Silent/Offstage Stakeholders** — не участвуют в шагах, но система защищает их интересы (владелец системы, регулятор, аудит).

Найти silent stakeholders → самые частые упущенные требования. Спроси: «Кто ещё пострадает, если UC провалится?»

---

### Scope (Область проектирования)

Всегда явно указывай scope в каждом UC:
- **Enterprise** (название компании) — бизнес-UC, белый ящик, описывает операции компании
- **System** (название системы) — требования к разрабатываемой системе, чёрный ящик
- **Subsystem** (название подсистемы) — редко, для документирования компонентов

**Инструменты определения scope:**
- In/Out List — таблица «входит/не входит», главное против scope creep
- Actor-Goal List — все акторы и их цели, первичный контракт о функциональности
- Use Case Briefs — 2–6 предложений на UC, достаточно для оценки сложности

## ЧАСТЬ 2 — СТРУКТУРА USE CASE (FULLY DRESSED TEMPLATE)

```
Use Case [ID]: [Имя — активный глагол + объект]
─────────────────────────────────────────────
Context of use:   [Краткий контекст ситуации]
Scope:            [Название системы/предприятия]
Level:            [Summary / User Goal / Subfunction]
Primary Actor:    [Кто инициирует]

Stakeholders & Interests:
  [Актор 1] — [интерес 1]
  [Актор 2] — [интерес 2]
  ...

Preconditions:
  [Что гарантированно верно до запуска — система не будет проверять это снова]

Minimal Guarantee:
  [Что система обещает ВСЕМ stakeholders даже при неудаче — обычно: залоггировано]

Success Guarantee:
  [Что выполнено при успехе — интересы всех stakeholders удовлетворены]

Trigger:
  [Событие, запускающее UC]

Main Success Scenario:
  1. [Актор делает X]
  2. [Система валидирует Y]
  3. [Система обновляет Z]
  ...

Extensions:
  [N][буква]. [Условие обнаружения]:
    [N][буква]1. [Шаг обработки]
    [N][буква]2. [Следующий шаг]
  ...

Technology & Data Variations List:
  [N][буква]. [Вариации КАК, не ЧТО — только список, не action steps]
```

## ЧАСТЬ 3 — ПРАВИЛА НАПИСАНИЯ ШАГОВ (10 Guidelines)

### Единственная форма предложения (Reminder 3)
```
[Актор]  [активный глагол]  [объект]  [опц. предложная группа].
Система  валидирует         PIN-код   от пользователя.
Студент  подаёт             заявку    на проект.
```
- Настоящее время, активный залог, субъект — первое или второе слово.
- Условие расширения — фраза (не предложение), завершается двоеточием: `Неверный PIN:`

### Кто владеет мячом? (Reminder 5)
- В каждом шаге должно быть ясно, **кто** выполняет действие.
- Избегай: «Данные сохраняются» → пиши: «Система сохраняет данные».

### Вид с высоты птичьего полёта (Guideline 3)
- Пиши от третьего лица наблюдателя, не от имени системы «изнутри».

### Шаг двигает процесс вперёд (Guideline 4)
- Если шаг не продвигает цель — объедини его с соседним.

### Намерение, не движения (Guideline 5 / Reminder 7)
- ❌ `Пользователь нажимает кнопку «Откликнуться»`
- ✅ `Студент подаёт заявку на участие в проекте`
- Никакого UI: нет кнопок, форм, полей, цветов — это работа UX-дизайнера.

### «Validates», не «checks» (Guideline 7)
- ❌ `Система проверяет, верен ли пароль`
- ✅ `Система валидирует пароль пользователя`
- Слово «validates» означает успех → неудача уходит в Extensions.

### Повторения и параллелизм (Guidelines 8, 10)
- «Студент повторяет шаги 3–4 до тех пор, пока не выберет все нужные проекты»
- «Шаги 2–4 можно выполнять в любом порядке»

## ЧАСТЬ 4 — РАСШИРЕНИЯ (EXTENSIONS)

### Три фазы работы с расширениями
1. **Brainstorm** — выписать ВСЕ возможные отклонения (не обрабатывать ещё)
2. **Rationalize** — убрать недетектируемые; объединить те, что дают одинаковый системный ответ
3. **Write** — написать обработку каждого

### Что включать в brainstorm
- Альтернативные пути к успеху («Клерк использует шорткод»)
- Неверные данные от primary actor
- Бездействие primary actor (timeout)
- Каждый «validates» в main → расширение для failure
- Нет ответа от supporting actor
- Внутренний сбой системы
- Аномальный внутренний сбой (corrupt data)
- Performance failure

### Формат расширения
```
[шаг][буква]. [Условие (что система обнаружила)]:
  [шаг][буква]1. [Что происходит]
  [шаг][буква]2. [Продолжение, если нужно]
```
- `*a.` — если условие может возникнуть **в любой момент**
- `2-5a.` — если условие может возникнуть **на шагах 2–5**
- Не писать «go to step N» без нужды — обычно очевидно

### Четыре варианта завершения расширения
1. Шаг исправлен, сценарий продолжается как будто успех (неявно — без «go to»)
2. Актор получает ещё один шанс (сценарий возвращается к тому же шагу)
3. UC завершается неудачей — «System notifies user, terminates session»
4. Альтернативный путь к успеху (без упоминания main scenario)

### Roll up failures
При ссылке на sub UC: если sub UC провалился — писать только одно расширение для failure, не все внутренние подпричины.

## ЧАСТЬ 5 — СВЯЗЫВАНИЕ USE CASES

### Sub Use Cases (includes)
- Шаг в main scenario просто называет другой UC: `Студент авторизуется в системе`
- Base UC знает о sub UC и когда его вызывать.
- По умолчанию использовать только этот тип связи.

### Extension Use Cases (extends)
- Использовать **только** в двух ситуациях:
  1. Асинхронные/прерывающие сервисы (Word processor: spell check прерывает typing)
  2. Добавление функционала к **залоченному** базовому UC
- Base UC **не знает** о расширяющем UC.
- Расширяющий UC сам указывает: в какой UC и при каком условии он вступает.

## ЧАСТЬ 6 — ПОРЯДОК РАБОТЫ (12-Step Recipe + Work Breadth First)

### 12-Step Recipe (Reminder 18)
```
1.  Найти границы системы (In/Out list)
2.  Brainstorm primary actors
3.  Brainstorm цели каждого актора (Actor-Goal List)
4.  Написать Summary use cases, покрывающие все цели
5.  Пересмотреть и скорректировать Summary UCs
6.  Выбрать один UC для детальной проработки
7.  Заполнить stakeholders, interests, preconditions, guarantees
8.  Написать main success scenario; проверить against guarantees
9.  Brainstorm и выписать все failure/extension conditions
10. Написать обработку каждого расширения
11. Вынести sub UCs, которым нужно своё место
12. Ревизия всего набора: дополнить, убрать, объединить, проверить полноту
```

### Порядок уточнения (Reminder 17 — Work Breadth First)
```
1. Акторы → 2. Цели → 3. Main success scenarios →
4. Список failure conditions (только список!) →
5. Recovery steps (обработка) → 6. Data fields → 7. Data-field details
```
⚠️ Не пиши обработку расширений до тех пор, пока не выписаны ВСЕ условия.

## ЧАСТЬ 7 — STAKEHOLDERS & GUARANTEES (Reminder 9)

Для каждого UC спросить:
- **Primary actor:** хочет получить [цель UC]
- **Владелец системы/компании:** хочет убедиться, что никто ничего не получил бесплатно / что всё залоггировано
- **Регулятор/аудит:** хочет видеть лог всех действий
- **Тот, кто должен восстановить систему:** хочет знать, на каком шаге всё остановилось

**Minimal Guarantee** = что выполнено при **любом** исходе (успех/неудача):
- Чаще всего: «Система залоггировала ход выполнения»

**Success Guarantee** = что выполнено при **успехе**:
- Найти: «Что разочарует stakeholder, если этого нет после успешного UC?»
- Написать отрицание этого разочарования.

## ЧАСТЬ 8 — 31 PASS/FAIL ТЕСТ (Reminder 11)

Для каждого написанного UC проверить — ответ на всё должен быть «Да»:

**Название:**
1. Активный глагол + цель primary actor?
2. Система может это доставить?

**Scope & Level:**
3. Поля Scope и Level заполнены?
4. Система из Scope трактуется как чёрный ящик?
5. Дизайнеры должны реализовать всё внутри Scope и ничего вне?
6. Содержание UC соответствует заявленному Level?
7. Цель действительно на этом уровне?

**Primary Actor:**
8. Названный primary actor имеет поведение (может выполнить «IF»)?
9. У него есть цель против SuD, которую система обещает выполнить?

**Preconditions:**
10. Они обязательны и гарантированы системой?
11. В UC они нигде не проверяются заново?

**Stakeholders & Interests:**
12. Они указаны?

**Minimal Guarantee:**
13. Интересы всех stakeholders защищены при неудаче?

**Success Guarantee:**
14. Интересы всех stakeholders удовлетворены при успехе?

**Main Success Scenario:**
15. Идёт от trigger до success end condition?
16. Последовательность шагов верна?
17. Шагов 3–9?

**Каждый шаг в любом сценарии:**
18. Сформулирован как цель, которая успешно выполнена?
19. После шага процесс заметно продвинулся вперёд?
20. Ясно, кто выполняет действие (кто «владеет мячом»)?
21. Намерение актора ясно?
22. Уровень цели шага ниже уровня цели всего UC?
23. Шаг не описывает UI-дизайн системы?
24. Ясно, какая информация передаётся?
25. Используется «validates» (не «checks»)?

**Extension condition:**
26. Система может и должна это обнаружить?
27. Система обязана это обработать?

**Technology/Data Variations List:**
28. Это точно не обычное поведенческое расширение?

**Весь UC:**
29. Спонсоры/пользователи: «Это то, что вы хотите?»
30. Спонсоры/пользователи: «По готовности системы вы сможете проверить, получили ли это?»
31. Разработчики: «Вы можете это реализовать?»

## ЧАСТЬ 9 — 8 ТИПИЧНЫХ ОШИБОК (из статьи «Руководство по Use Cases»)

| # | Ошибка | Симптом | Исправление |
|---|--------|---------|-------------|
| 1 | Только действия пользователя, без ответа системы | «Пользователь вводит PIN» без «Система проверяет...» | На каждый значимый шаг пользователя — реакция системы |
| 2 | Не указан актор (обезличенные шаги) | «Получить PIN-код. Выдать наличные.» | Явно: «Система выдаёт наличные» / «Клиент вводит PIN» |
| 3 | Чрезмерная UI-детализация | Описывает каждое поле, кнопку, цвет | Убрать UI, оставить намерение: «Студент вводит данные заявки» |
| 4 | Слишком мелкая атомарность шагов | 3 отдельных шага для имени, адреса, телефона | Объединить в «Пользователь вводит персональные данные» |
| 5 | Пропущены альтернативные сценарии | Только happy path | Задать «что если нет?» на каждом шаге |
| 6 | Нечёткие preconditions и triggers | UC начинается ниоткуда | Явно: «Предусловие: пользователь авторизован» |
| 7 | Нарушение «чёрного ящика» | Внутренние технические шаги (API calls, SQL) | Описывать внешнее поведение системы, не реализацию |
| 8 | Несколько целей в одном UC | UC слишком большой, нелинейный | Один UC = одна цель primary actor |

## ЧАСТЬ 10 — ТЕХНОЛОГИИ И ВАРИАЦИИ ДАННЫХ

Раздел «Technology & Data Variations List» — только если способ достижения цели может варьироваться:
- **Что** — одинаково → цель та же
- **Как** — разное → Variations List

Формат: только список, **никаких action steps**.
```
Technology & Data Variations List:
  2a. Идентификация: банковская карта, скан сетчатки или отпечаток пальца.
  7a. Оплата: чек, EFTS, или кредит на следующую покупку.
```

## ЧАСТЬ 11 — ФОРМАТЫ И СТИЛИ

| Формат | Когда использовать |
|--------|--------------------|
| **Fully Dressed** | Критичные UC, распределённая команда, фиксированный контракт |
| **Casual** | Маленькая команда, ранний этап, высокая коммуникация |
| **RUP style** | Корпоративные стандарты |
| **Two-column** | При проектировании UI (не для требований) |

**Принцип «Blue Jeans»:** Лучше написать меньше на более высоком уровне — тогда документ прочтут и зададут вопросы. Подробный неправильный UC хуже, чем краткий правильный.

**Use Case Diagram** — только таблица содержания (контекстная диаграмма). Не заменяет текст. Эллипсы без текста — ничего не значат.

## ЧАСТЬ 12 — USER STORY vs USE CASE (сводка из статей)

| | User Story | Use Case |
|---|-----------|----------|
| Формат | «Как [роль], я хочу [действие], чтобы [ценность]» | Structured template (Cockburn) |
| Объём | 1–2 строки | Полный сценарий с расширениями |
| Фокус | Результат/фича для пользователя | Процесс достижения результата |
| Используется | Agile, изменяемые требования | Фиксированный объём, сложные сценарии |
| Альтернативы | Критерии приёмки | Extensions section |

**INVEST-критерии для User Stories:** Independent, Negotiable, Valuable, Estimable, Small, Testable.

**Практика:** US = заголовок + контекст; UC = детальное описание как реализовать эту историю.

## ЧАСТЬ 13 — КРАТКОЕ ПРИМЕНЕНИЕ К ПРОЕКТУ DIGITAL STUDENT ASSISTANT

### Предварительный Actor-Goal List (из BPMN)

| Актор | Цели (user goal level) |
|-------|----------------------|
| Студент | Найти проект, Подать заявку на проект |
| Преподаватель | Создать проект, Рассмотреть заявку студента |
| ЦППРП | Промодерировать проект |

### Система (Scope)
Digital Student Assistant — система, которую мы проектируем. Черный ящик.

### Ключевые заметки по проекту
- Шлюзы в BPMN = Extension conditions в Use Cases
- MessageFlow в BPMN = взаимодействие между акторами в steps

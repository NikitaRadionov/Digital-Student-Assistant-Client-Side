# UC-09: Создать проект

| Атрибут | Значение |
|---|---|
| **Scope** | Цифровой Студенческий Ассистент |
| **Level** | 🌊 Пользовательская цель (sea-level) |
| **Primary Actor** | Преподаватель (Заказчик) |
| **Supporting Actors** | Система (создаёт черновик, сохраняет данные) |
| **Revision** | 1.0 · 2026-05-09 |

---

## Цель в контексте

Преподаватель хочет сформировать описание своего проекта и сохранить его как черновик — чтобы впоследствии доработать и опубликовать для студентов.

> **Примечание:** Создание черновика — первый шаг двухшагового процесса публикации. Публикация завершается в [UC-10: Отправить проект на модерацию](UC-10-Submit-For-Moderation.md) → [UC-12: Промодерировать проект](UC-12-Moderate-Project.md).

---

## Стейкхолдеры и интересы

| Стейкхолдер | Интерес |
|---|---|
| Преподаватель | Быстро зафиксировать идею проекта; иметь возможность доработать перед публикацией |
| Студенты | Видеть в каталоге качественно описанные проекты с реальными требованиями |
| ЦППРП | Получать на модерацию проекты, соответствующие минимальным требованиям |
| Система | Сохранить проект с валидными данными; установить корректный `source_type` и начальный статус |

---

## Предусловия

1. Преподаватель авторизован с ролью `CUSTOMER`

> Авторизация не является предусловием — обрабатывается в расширении **1а**.

---

## Триггер

Преподаватель нажимает «Создать проект» в личном кабинете или в каталоге проектов.

---

## Гарантии

### Минимальная гарантия
При любом исходе (ошибка валидации, отказ) никаких записей `Project` в БД не создаётся. Форма возвращается с сообщением об ошибке.

### Гарантия успеха
- Запись `Project` создана: `status = DRAFT`, `source_type = SUPERVISOR`, `owner = преподаватель`
- Сохранены: `title`, `description`, `tech_tags`, `team_size`
- Преподаватель перенаправлен на детальную страницу созданного проекта
- Уведомление «Проект создан!»

---

## Основной сценарий успеха

| Шаг | Кто | Действие |
|---|---|---|
| 1 | Преподаватель | Открывает форму создания проекта |
| 2 | Система | Отображает форму создания проекта |
| 3 | Преподаватель | Вводит название проекта, описание, технологические теги и размер команды |
| 4 | Преподаватель | Отправляет форму |
| 5 | Система | Валидирует данные формы → [BR-PRJ-01], [BR-PRJ-04], [TV-1] |
| 6 | Система | Создаёт запись `Project` со статусом `DRAFT` и `source_type = SUPERVISOR` |
| 7 | Система | Перенаправляет преподавателя на детальную страницу проекта |
| 8 | Система | Отображает уведомление «Проект создан!» |

---

## Расширения

### 1а. Преподаватель не авторизован

- 1а.1. Система перенаправляет на `/auth/?next=/projects/create/`
- 1а.2. Преподаватель проходит авторизацию (включить [UC-02](UC-02-Login.md))
- → Продолжить с шага 1

---

### 1б. Пользователь авторизован, но роль ≠ CUSTOMER
> Студент или ЦППРП пытаются открыть форму создания проекта. [BR-PRJ-01]

- 1б.1. Система возвращает ошибку доступа (HTTP 403)
- **Сценарий завершается**, форма не отображается.

---

### 5а. Название не указано или превышает 255 символов

- 5а.1. Система возвращает форму с ошибкой: «Название обязательно» / «Не более 255 символов»
- → Вернуться к шагу 3

---

### 5б. Размер команды вне допустимого диапазона (< 1 или > 100)
> [BR-PRJ-08]

- 5б.1. Система возвращает форму с ошибкой: «Минимум 1 участник» / «Максимум 100 участников»
- → Вернуться к шагу 3

---

### 5в. Описание превышает 2000 символов

- 5в.1. Система возвращает форму с ошибкой: «Описание не может превышать 2000 символов»
- → Вернуться к шагу 3

---

### 5г. Технологические теги не прошли валидацию
> Недопустимые символы, тег длиннее 50 символов или суммарно больше 20 тегов.

- 5г.1. Система возвращает форму с ошибкой: «Недопустимые теги: ...»
- → Вернуться к шагу 3

---

### 3а. Преподаватель не указывает описание (оставляет поле пустым)
> Описание необязательно для проектов преподавателя. [BR-PRJ-09]

- 3а.1. Система сохраняет `description = ""`
- → Продолжить с шага 4 (поведение штатное, не исключение)

## Переходы состояний после создания

```
Project: DRAFT
  │
  ├─► ON_MODERATION   — преподаватель отправляет на модерацию → UC-10
  │     │
  │     ├─► PUBLISHED         — ЦППРП одобряет → UC-12
  │     └─► REJECTED          — ЦППРП отклоняет → UC-12 [terminal]
  │           └─► (DRAFT)     — преподаватель может создать новый проект
  │
  └─► [deleted]        — преподаватель удаляет черновик (только DRAFT/REJECTED)
```

**Сравнение с инициативным проектом (UC-06):**

| Аспект | UC-09 (Supervisor) | UC-06 (Initiative) |
|---|---|---|
| Актор | Преподаватель | Студент |
| Начальный статус | `DRAFT` | `ON_MODERATION` |
| Поле `supervisor_name` | нет | есть (необязательное) |
| `description` | необязательно | обязательно |
| `team_size` max | 100 | 10 |
| Отказ при модерации | → `REJECTED` (terminal) | → `REVISION_REQUESTED` (цикл) |

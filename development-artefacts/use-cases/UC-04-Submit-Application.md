# UC-04: Подать заявку на проект

| Атрибут | Значение |
|---|---|
| **Scope** | Цифровой Студенческий Ассистент |
| **Level** | 🌊 Пользовательская цель (sea-level) |
| **Primary Actor** | Студент |
| **Supporting Actors** | Система (создаёт запись, обновляет UI-компонент) |

---

## Цель в контексте

Студент хочет зафиксировать свой интерес к проекту и передать заявку преподавателю на рассмотрение — чтобы в итоге попасть в команду проекта.

---

## Стейкхолдеры и интересы

| Стейкхолдер | Интерес |
|---|---|
| Студент | Быстро и без дублей отправить заявку; получить подтверждение |
| Преподаватель (Заказчик) | Получить заявку с мотивацией, достаточной для принятия решения |
| ЦППРП | Корректная статистика заявок по всей платформе |
| Система | Обеспечить уникальность заявки ([BR-APP-01](#)), консистентность счётчиков ([BR-STA-01](#)) |

---

## Предусловия

1. Проект существует в системе

> Авторизация и состояние проекта — не предусловия: обрабатываются в расширениях **1а**, **5а**, **5б**.

---

## Триггер

Студент нажимает «Откликнуться» на карточке проекта в каталоге  
**или** «Подать заявку» на детальной странице проекта.

---

## Гарантии

### Минимальная гарантия
Дублирующая заявка не создаётся. Данные студента, проекта и существующих заявок не изменяются ни при каком исходе сценария.

### Гарантия успеха
- Заявка создана со статусом «На рассмотрении»; сохранены данные студента, проекта и мотивационный текст
- Студент получает уведомление «Заявка успешно отправлена!»
- Заявка отображается во вкладке «Мои заявки» с обновлёнными счётчиками

---

## Основной сценарий успеха

| Шаг | Кто | Действие |
|---|---|---|
| 1 | Студент | Открывает детальную страницу проекта из каталога |
| 2 | Студент | Инициирует подачу заявки на проект |
| 3 | Система | Отображает форму подачи заявки |
| 4 | Студент | Вводит текст мотивации и подтверждает подачу |
| 5 | Система | Валидирует, что проект принимает заявки: статус `PUBLISHED` и есть свободные места → [BR-APP-02] |
| 6 | Система | Валидирует отсутствие активной заявки данного студента на этот проект → [BR-APP-01] |
| 7 | Система | Создаёт заявку со статусом «На рассмотрении» |
| 8 | Система | Уведомляет студента об успешной подаче заявки |

---

## Расширения

### 1а. Студент не авторизован
> Обнаруживается при первом обращении к любому из endpoints подачи заявки.

- 1а.1. Система перенаправляет на `/auth/?next=/projects/<pk>/`
- 1а.2. Студент проходит авторизацию (включить [UC-02](UC-02-Login.md))
- 1а.3. → Продолжить с шага 2

---

### 1б. Пользователь авторизован, но роль ≠ STUDENT
> Преподаватель или ЦППРП нажали кнопку «Откликнуться». [BR-APP-03]

- 1б.1. Система возвращает HTTP 400: «Подавать заявки могут только студенты»
- **Сценарий завершается**, заявка не создаётся.

---

### 2а. Студент использует быстрый отклик с карточки каталога (без мотивационного письма)
> Альтернативный entry point — `apply_to_project` view. [TV-1]

- 2а.1. Система пропускает шаги 3–4 (поле мотивации в этом пути отсутствует)
- 2а.2. Мотивационный текст сохраняется как пустая строка
- → Продолжить с шага 5

---

### 5а. Проект изменил статус с момента загрузки страницы (status ≠ PUBLISHED)
> Например, проект снят с публикации между загрузкой каталога и отправкой формы.

- 5а.1. Система возвращает HTTP 400: «Project is not accepting applications»
- **Сценарий завершается**, заявка не создаётся.

---

### 5б. Все места в проекте заполнены (accepted_participants_count ≥ team_size)
> Race condition: места могли закончиться в промежутке между просмотром каталога и отправкой.

- 5б.1. Система возвращает HTTP 400: «Project is not accepting applications»
- **Сценарий завершается**, заявка не создаётся.

---

### 6а. Студент уже подавал заявку на этот проект
> [BR-APP-01]: UniqueConstraint не допускает повторную заявку.

- 6а.1. Система обнаруживает существующую заявку этого студента на данный проект
- 6а.2. Система уведомляет: «Вы уже подавали заявку на этот проект»
- 6а.3. Существующая заявка сохраняется без изменений
- **Сценарий завершается**, дубль не создаётся.

---

## Перечень технологических и пользовательских вариаций

| # | Переменная часть | Варианты |
|---|---|---|
| TV-1 | Путь подачи | **(A)** детальная страница + мотивационное письмо (`submit_application`, `POST /projects/<pk>/submit-application/`); **(B)** быстрый отклик с карточки каталога без письма (`apply_to_project`, `POST /projects/<pk>/apply/`) |
| TV-2 | Транспорт ответа | **(A)** HTMX-запрос — сервер возвращает HTML-partial, кнопка заменяется inline; **(B)** `fetch()`-вызов от неавторизованного — JSON `{"error": "unauthenticated", "redirect": "/auth/"}` с HTTP 401 |
| TV-3 | Мотивационный текст | **(A)** непустой текст произвольной длины; **(B)** пустая строка — принимается при первичной подаче ([BR-APP-04]); при редактировании минимум 30 символов ([BR-APP-05]) |

## Бизнес-правила

> Полный справочник в [00-business-rules.md](00-business-rules.md)

| ID | Правило |
|---|---|
| [BR-APP-01](00-business-rules.md#заявки-applications) | Уникальность: одна запись `Application` на пару `(project_id, applicant_id)` — UniqueConstraint на уровне БД |
| [BR-APP-02](00-business-rules.md#заявки-applications) | Проект принимает заявки только при: `status == PUBLISHED` AND `accepted_participants_count < team_size` |
| [BR-APP-03](00-business-rules.md#заявки-applications) | Право подачи — только роль `STUDENT` (`is_staff = False`) |
| [BR-APP-04](00-business-rules.md#заявки-applications) | Мотивационный текст при первичной подаче необязателен |
| [BR-APP-05](00-business-rules.md#заявки-applications) | При редактировании существующей заявки мотивация ≥ 30 символов |

---

## Переходы состояний после создания заявки

```
Application: SUBMITTED
  │
  ├─► ACCEPTED  — преподаватель принял   → UC-11
  ├─► REJECTED  — преподаватель отклонил → UC-11 (comment ≥ 50 chars)
  └─► [deleted] — студент отозвал        → UC-05
```

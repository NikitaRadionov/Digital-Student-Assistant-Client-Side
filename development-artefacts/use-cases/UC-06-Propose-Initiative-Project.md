# UC-06: Предложить инициативный проект

| Атрибут | Значение |
|---|---|
| **Scope** | Цифровой Студенческий Ассистент |
| **Level** | 🌊 Пользовательская цель (sea-level) |
| **Primary Actor** | Студент |
| **Supporting Actors** | Система (создаёт проект, помещает в очередь модерации) |
| **Revision** | 1.0 · 2026-05-09 |

---

## Цель в контексте

Студент хочет предложить собственную идею проекта — чтобы после одобрения ЦППРП найти команду и/или руководителя среди преподавателей.

> **Отличие от UC-09:** создатель — Студент, а не Преподаватель; проект сразу поступает на модерацию без стадии черновика; при отклонении ЦППРП возвращает проект на доработку (REVISION_REQUESTED), а не отклоняет окончательно.

---

## Стейкхолдеры и интересы

| Стейкхолдер | Интерес |
|---|---|
| Студент | Быстро зафиксировать и передать идею на проверку; знать результат модерации |
| ЦППРП | Получать достаточно описанные проекты; иметь возможность вернуть на доработку без финального отказа |
| Преподаватель (желаемый руководитель) | Получить уведомление о предложении (за рамками текущей реализации) |
| Система | Сохранить `source_type = INITIATIVE`; установить `status = ON_MODERATION` при создании |

---

## Предусловия

1. Студент авторизован с ролью `STUDENT`

> Авторизация не является предусловием — обрабатывается в расширении **1а**.

---

## Триггер

Студент переходит по ссылке «Предложить проект» и открывает форму инициативного проекта (`/projects/initiative/`).

---

## Гарантии

### Минимальная гарантия
При ошибке валидации никаких записей `Project` в БД не создаётся. Форма возвращается с сообщениями об ошибках.

### Гарантия успеха
- Запись `Project` создана: `status = ON_MODERATION`, `source_type = INITIATIVE`, `owner = студент`
- Сохранены: `title`, `description`, `tech_tags`, `team_size`, `supervisor_name`
- Студент перенаправлен на детальную страницу созданного проекта
- Уведомление «Проект отправлен на проверку!»
- Проект появился в очереди ЦППРП (UC-12)

---

## Основной сценарий успеха

| Шаг | Кто | Действие |
|---|---|---|
| 1 | Студент | Открывает форму инициативного проекта |
| 2 | Система | Отображает форму создания инициативного проекта |
| 3 | Студент | Вводит название, описание проекта, технологические теги, желаемый размер команды и (опционально) имя желаемого руководителя |
| 4 | Студент | Отправляет форму |
| 5 | Система | Валидирует данные формы → [BR-PRJ-02], [BR-PRJ-03], [BR-PRJ-08], [BR-PRJ-09] |
| 6 | Система | Создаёт запись `Project`: `status = ON_MODERATION`, `source_type = INITIATIVE` |
| 7 | Система | Перенаправляет студента на детальную страницу проекта |
| 8 | Система | Отображает уведомление «Проект отправлен на проверку!» |

---

## Расширения

### 1а. Студент не авторизован

- 1а.1. Система перенаправляет на `/auth/?next=/projects/initiative/`
- 1а.2. Студент проходит авторизацию (включить [UC-02](UC-02-Login.md))
- → Продолжить с шага 1

---

### 1б. Пользователь авторизован, но роль ≠ STUDENT
> [BR-PRJ-02]: декоратор `@student_required`.

- 1б.1. Система возвращает HTTP 403 Forbidden
- **Сценарий завершается**.

---

### 5а. Название не указано или превышает 255 символов

- 5а.1. Система возвращает форму с ошибкой: «Название обязательно» / «Не более 255 символов»
- → Вернуться к шагу 3

---

### 5б. Описание не указано
> [BR-PRJ-09]: для инициативных проектов `description` является **обязательным** полем (в отличие от UC-09).

- 5б.1. Система возвращает форму с ошибкой: «Опишите проект — это главное поле»
- → Вернуться к шагу 3

---

### 5в. Описание превышает 2000 символов

- 5в.1. Система возвращает форму с ошибкой: «Описание не может превышать 2000 символов»
- → Вернуться к шагу 3

---

### 5г. Размер команды вне допустимого диапазона (< 1 или > 10)
> [BR-PRJ-08]: для инициативных проектов `max_value = 10` (вдвое меньше, чем у supervisor).

- 5г.1. Система возвращает форму с ошибкой: «Минимум 1 участник» / «Максимум 10 участников»
- → Вернуться к шагу 3

---

### 5д. Технологические теги не прошли валидацию
> Недопустимые символы, тег длиннее 50 символов или суммарно больше 20 тегов.

- 5д.1. Система возвращает форму с ошибкой: «Недопустимые теги: ...»
- → Вернуться к шагу 3

---

### После создания: ЦППРП отклоняет проект → REVISION_REQUESTED
> Расширяет UC-12: при `source_type = INITIATIVE` отклонение возвращает статус `REVISION_REQUESTED`, а не `REJECTED`. [BR-MOD-05]

- Студент получает уведомление о результате (при следующем посещении страницы проекта)
- Студент видит статус «На доработке» и комментарий ЦППРП
- Студент редактирует проект (поля title / description / tags / team_size — `project_edit` view)

> ⚠️ **Известное ограничение реализации:** переход `REVISION_REQUESTED → ON_MODERATION` через `submit_project_for_moderation` требует роли `CUSTOMER`, которой у студента нет. Повторная отправка инициативного проекта на модерацию в текущей версии системы не поддерживается на уровне UI. Статус `REVISION_REQUESTED` зафиксирован в модели и transitions, но соответствующая кнопка для студента отсутствует.

## Сравнение с UC-09 «Создать проект»

| Аспект | UC-06 (Initiative) | UC-09 (Supervisor) |
|---|---|---|
| Первичный актор | Студент | Преподаватель |
| Начальный статус | `ON_MODERATION` (сразу) | `DRAFT` (черновик) |
| Поле `description` | **Обязательно** | Необязательно |
| Поле `supervisor_name` | Есть (опционально) | Нет |
| `team_size` max | **10** | 100 |
| Отдельный шаг UC-10 | Нет | Да |
| При отклонении ЦППРП | → `REVISION_REQUESTED` (возвратный) | → `REJECTED` (терминальный) |

| # | Переменная часть | Варианты |
|---|---|---|
| TV-1 | `supervisor_name` | **(A)** заполнено — пожелание по руководителю; **(B)** пустая строка — допустимо, проект без указанного руководителя |

                              ┌─────────────────┴──────────────────┐
                           approve                               reject
                              │                                    │

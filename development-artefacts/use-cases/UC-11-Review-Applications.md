# UC-11: Рассмотреть заявки на проект

| Атрибут | Значение |
|---|---|
| **Scope** | Цифровой Студенческий Ассистент |
| **Level** | 🌊 Пользовательская цель (sea-level) |
| **Primary Actor** | Преподаватель (Заказчик) |
| **Supporting Actors** | Система (обновляет статус заявки, пересчитывает укомплектованность) |
| **Revision** | 1.0 · 2026-05-09 |

---

## Цель в контексте

Преподаватель хочет просмотреть поступившие заявки, оценить кандидатов и принять решение по каждому — чтобы сформировать команду проекта из подходящих студентов.

---

## Стейкхолдеры и интересы

| Стейкхолдер | Интерес |
|---|---|
| Преподаватель | Видеть все заявки в одном месте; быстро принять обоснованное решение по каждому студенту |
| Студент | Получить решение (принят / отклонён) с понятным комментарием при отказе |
| ЦППРП | Корректная статистика принятых студентов по платформе |
| Система | Поддерживать консистентность счётчика принятых участников; автоматически переводить проект в «Укомплектован» при заполнении мест |

---

## Предусловия

1. Преподаватель авторизован

---

## Триггер

Преподаватель переходит на страницу заявок своего проекта (`/projects/<pk>/applications/`).

---

## Гарантии

### Минимальная гарантия
Статус заявки не изменяется при ошибке валидации (короткий комментарий при отказе) или нехватке прав. Счётчик принятых участников остаётся консистентным.

### Гарантия успеха (принятие)
- Заявка переходит в статус «Принята»; данные проверяющего и время решения сохранены
- Счётчик принятых участников проекта обновлён
- При заполнении всех мест: проект автоматически переходит в статус «Укомплектован» → [BR-STA-02]
- Студент уведомлён (при следующем посещении системы)

### Гарантия успеха (отклонение)
- Заявка переходит в статус «Отклонена»; комментарий и данные проверяющего сохранены
- Студент видит статус «Отклонена» и комментарий в своём списке заявок

---

## Основной сценарий успеха (принятие заявки)

| Шаг | Кто | Действие |
|---|---|---|
| 1 | Преподаватель | Открывает страницу заявок своего проекта |
| 2 | Система | Отображает список заявок, сгруппированных по статусу; показывает счётчик свободных мест |
| 3 | Преподаватель | Выбирает заявку для рассмотрения |
| 4 | Система | Отображает заявку: мотивационный текст и профиль студента |
| 5 | Преподаватель | Принимает решение принять заявку |
| 6 | Система | Валидирует права актора и статус заявки → [BR-APP-06] |
| 7 | Система | Обновляет заявку: статус переходит в «Принята»; сохраняет данные проверяющего и время решения |
| 8 | Система | Пересчитывает укомплектованность проекта; перенаправляет с уведомлением «Заявка принята!» |

---

## Расширения

### 1а. Преподаватель не авторизован

- 1а.1. Система перенаправляет на `/auth/?next=/projects/<pk>/applications/`
- 1а.2. Преподаватель проходит авторизацию (включить [UC-02](UC-02-Login.md))
- → Продолжить с шага 1

---

### 1б. Пользователь не является владельцем проекта
> [BR-APP-06]: управлять заявками может только владелец проекта.

- 1б.1. Система возвращает HTTP 404 (скрывает сам факт существования страницы)
- **Сценарий завершается**.

---

### 5б. Преподаватель принимает решение «Отклонить»

- 5б.1. Преподаватель вводит комментарий с обоснованием отказа
- 5б.2. Система валидирует длину комментария ≥ 50 символов → [BR-APP-07]
  - **Если комментарий короче 50 символов:** возвращает ошибку «Комментарий при отклонении должен содержать не менее 50 символов»; → Вернуться к шагу 5б.1
- 5б.3. Система обновляет заявку: статус переходит в «Отклонена»; сохраняет комментарий и данные проверяющего
- 5б.4. Система пересчитывает укомплектованность проекта (счётчик не изменяется)
- 5б.5. Система перенаправляет с уведомлением «Заявка отклонена»
- **Сценарий завершается**.

---

### 6а. Заявка уже обработана (status ≠ SUBMITTED) — параллельное изменение
> Например, студент отозвал заявку (UC-05) в промежутке между просмотром и отправкой решения.

- 6а.1. Система сообщает об ошибке: «Заявку можно рассматривать только в статусе "На рассмотрении"»
- → Вернуться к шагу 2 (список обновляется)

---

### 8а. После принятия заявки все места в проекте заполнены
> [BR-STA-02]: система автоматически переводит проект в статус «Укомплектован».

- 8а.1. Система обнаруживает, что количество принятых участников достигло размера команды
- 8а.2. Проект автоматически переходит в статус «Укомплектован»; новые заявки перестают приниматься
- → Продолжить с шага 8 (штатное поведение; преподаватель видит обновлённый статус)

---

### 2а. Преподаватель фильтрует заявки по статусу
> Дополнительный сценарий навигации, не меняющий основной flow.

- 2а.1. Преподаватель выбирает фильтр: `SUBMITTED` / `ACCEPTED` / `REJECTED`
- 2а.2. Система отображает отфильтрованный список
- → Продолжить с шага 3

## Перечень технологических и пользовательских вариаций

| # | Переменная часть | Варианты |
|---|---|---|
| TV-1 | Решение | **(A)** `decision="accept"` → `ACCEPTED`; **(B)** `decision="reject"` → `REJECTED` (комментарий обязателен ≥ 50 символов) |
| TV-2 | Фильтрация списка | По параметру `?status=submitted/accepted/rejected`; пагинация по PAGE\_SIZE |
| TV-3 | Наличие мотивационного письма | **(A)** непустое — позволяет оценить кандидата; **(B)** пустое — заявка подана через quick-apply; допустимо |

## Переходы состояний после решения

```
Application: SUBMITTED
  │
  ├──[accept]──► ACCEPTED
  │                  │
  │     (auto: пересчёт укомплектованности)
  │                  │
  │        принятых ≥ team_size?
  │               YES → Project: STAFFED
  │                NO → Project: (остаётся PUBLISHED)
  │
  └──[reject]──► REJECTED
                    (комментарий сохранён; счётчик не изменяется)
```
